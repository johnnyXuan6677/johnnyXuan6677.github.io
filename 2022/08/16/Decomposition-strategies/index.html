第二章 Decomposition strategies(自己看板本)

分解策略

章節大綱

  • 深入了解軟體工程和其重要性
  • 利用decomposition patterns將系統細分出business capability進而取得各個服務的關聯架構
  • 使用bounded context 和DDD(domain driven design)解析故事讓decomposition 更加簡單

微服務架構到底是什麼東西?

  • 功能性分解(functional decomposition)
    • 過往的傳統架構喜歡將系統全部寫在一起(big ball of mud)
    • 功能性分解是一種不同於傳統架構設計風格，每個服務只鎖定處理某服務，將系統一一分離來避免麵條式代碼 (Spaghetti code)
    • 但是相對的這也會有許多挑戰和缺點
  • 軟體工程非常非常重要
    • 架構設計會很大的影響到系統品質(quality)
    • scalability(可擴充性)，reliability(可靠性)，security，maintainability，testability，and deployability
      • scalability–如何設計系統讓系統在擴充功能或是應用範圍變大的時候變得簡單
      • reliability–是指在給定時間內，特定環境下軟體無錯運行的概率。
      • maintainability
      • testability
      • deployability
    • 分解(decomposition)的兩種原因
      • 讓多數人或是團體可以快速並附有成效的理解軟體的內容物。(loose couple)
      • 可清楚定義軟體中各個元素的關聯

The software architecture of a computing system is the set of structures needed to reason about the system, which comprise software elements, relations among them, and properties of both. Documenting Software Architectures — Bass et al.

THE 4+1 VIEW MODEL OF SOFTWARE ARCHITECTURE

  • 相同於建築架構軟體也是需要一種設計樣式模型，Phillip Krutchen提出了4+1 view model詳細說明五種分析軟體架構的面相
  • logical view (邏輯觀點): 代表功能需求（系統應提供哪些服務給使用者）被拆解之後的結果。在一般軟體專案中若採用物件導向的方式開發，則Logic View以物件（object）或類別（class）和物件之間的關係（association、aggregation、usage、inheritance、instantiation）來呈現
  • Implementation view (實體觀點):將軟體對應到硬體，關注可用性、可靠性、效能、可擴展性等非功能需求。
  • Process view(行程觀點): 考慮效能或可用性這類的非功能需求，主要關注並行處理、分散式、資料與狀態完整性、容錯處理，以及如何將Logic View的靜態結構對應到Process View的動態視角。
  • Deployment view(開發觀點):將系統分解成子系統，軟體被打包成可以被個人或開發團隊所實作的程式庫、類別庫或子系統。子系統被組成階層式架構，每一層都有清楚定義的介面與上層子系統溝通。
  • Scenario View(腳本觀點):將其他四個觀點全部串連起來。
  • https://teddy-chen-tw.blogspot.com/2017/01/4-1-view.html

    Overview of architectural styles

    THE LAYERED ARCHITECTURAL STYLE

  • Presentation layer—接收外部請求( view、app、ap 等等 )，呼叫業務層，並將業務層回傳的 DTO(Data Transfer Object) 轉成 viewmodel 回傳給外部。
    • Contains code that implements the user interface or external APIs
  • Business logic layer—接收展示層請求，專注處理商業邏輯，呼叫資料層，並將資料層回傳的 model 轉換成 DTO 回傳給展示層。
    • Contains the business logic
  • Persistence layer—專注處理資料，資料來源來自資料庫或其他其他服務，接收業務層呼叫，執行資料處理回傳 model。
    • Implements the logic of interacting with the database
  • drawback
    • Single presentation layer—應用程式是有可能會需要調用超過一個單一系統資料的情況發生???????????????????????????
    • Single persistence layer—應用程式是有可能會需要與多個資料庫作互動
    • business logic 會強烈依賴於單一儲存層，造成你將無法將business logic和Database 分開測試。
      • Defines the business logic layer as depending on the persistence layer—In theory, this dependency prevents you from testing the business logic without the database.

        THE HEXAGONAL ARCHITECTURE STYLE

  • HEXAGONAL ARCHITECTURE是一種將商業邏輯放在中心的設計風格
  • Duncan Nisbet 用六角形的桌子來比喻六角架構
    • 所有的 domain objects 都在桌上。
    • 環繞桌子周圍的椅子就是 adapters。
    • 站在椅背後面的人等同於外部系統或服務。
    • 桌子的結構基本上不會變動（領域物件不變）；會變動的部分是跟外界銜接的 adapters。
    • 椅子的規格必須符合桌子的 port，才能卡進去－－adapters 必須符合 domain objects 的 port，彼此才能銜接得上。
  • 內六角是領域邏輯，外六角是 adapters，作為內六角與外部系統銜接的橋樑。六角架構的一個重點是，domain model 並不依賴任何 layer（例如資料存取層），而是所有的 layers 都依賴 domain model。
  • https://www.huanlintalk.com/2012/10/designing-layered-application-3-onion.html

Defining an application’s microservice architecture

  • 第一步驟:把故事化為運作流程
  • 第二步驟:分離並定義服務
  • 第三步驟:分析各個服務關聯性並設計service’s API

    Identifying the system operations

  • The First step:將故事描述成一個high-level domain model裡面包含key classes(Nouns)這些classes可以描述整個運作系統。
  • The Second step: 將故事裡的動作(verbs)分析成domain model並加以描述動作後actors的關聯性
  • Create a high-level domain model
    • A domain model is created using standard techniques such as analyzing the nouns in the stories and scenarios and talking to the domain experts
    • example :Place Order story
    • Nouns: Consumer,order,restaurant,and CreditCard
    • Accept Order story
    • Courier and Delivery classes,最後補足 MenuItem and Address
  • Defined System Operations
    • A good starting point for identifying system commands is to analyze the verbs in the user stories and scenarios.

    • Defined commands operations

  • Query command 省略

    Defining services by applying the Decompose by business capability pattern

    http://microservices.io/patterns/decomposition/decompose-by-business-capability.html
  • An organization’s business capabilities capture what an organization’s business is
  • 定義 business capabilities ???
    • example:
      • online store business capabilities
        • Order management
        • inventory management
        • shipping…and so on
  • 作者對 FTGO 的 business capabilities
    • Supplier management
      • Courier management—Managing courier information
      • Restaurant information management—Managing restaurant menus and other information, including location and open hours
    • Consumer management—Managing information about consumers
    • Order taking and fulfillment
      • Order management—Enabling consumers to create and manage orders
      • Restaurant order management—Managing the preparation of orders at a restaurant
      • Logistics
      • Courier availability management—Managing the real-time availability of couriers to delivery orders
      • Delivery management—Delivering orders to consumers
    • Accounting
      • Consumer accounting—Managing billing of consumers
      • Restaurant accounting—Managing payments to restaurants
      • Courier accounting—Managing payments to couriers
  • 寫出所有的business capilities後將其化為服務

    Defining service by applying the Decompose by sub-domain pattern

  • bounded context的想法就是不要把一整個大系統視為一個context，而是依據「某種方式」把整個大系統分成若干個bounded context（有界限的context）。
  • 設立bounded context並沒有一定的標準，純看當下的需求，通常會從語意(Linguistic)與業務能力(Business Capability)下手

    Decomposition guidelines

    SINGLE RESPONSIBILITY PRINCIPLE(SRP)

    A class should have only one reason to change. —Robert C. Martin

  • 一個模組應只對唯一一個腳色(actor)或利益相關者負責。
  • https://medium.com/%E7%A8%8B%E5%BC%8F%E6%84%9B%E5%A5%BD%E8%80%85/%E4%BD%BF%E4%BA%BA%E7%98%8B%E7%8B%82%E7%9A%84-solid-%E5%8E%9F%E5%89%87-%E5%96%AE%E4%B8%80%E8%81%B7%E8%B2%AC%E5%8E%9F%E5%89%87-single-responsibility-principle-c2c4bd9b4e79
  • https://medium.com/@f40507777/%E5%96%AE%E4%B8%80%E8%81%B7%E8%B2%AC%E5%8E%9F%E5%89%87-single-responsibility-principle-7b4eb03f1fff

    COMMON CLOSURE PRINCIPLE(CCP)

    The classes in a package should be closed together against the same kinds of changes. A change that affects a package affects all the classes in that package.
    —Robert C. Martin

  • 將會在相同時間 因為相同理由而發生變化的類別放在一起，最小化code複雜度

    Obstacles to decomposing an application into services

    Network latency

  • 分散式系統永遠的痛
  • 分離後的多個服務系統將會產生大量的溝通過程(process)造成延遲

    Reduced availability due to synchronous communication

  • 如何實現服務和服務之間的溝通又不會降低可用性
  • Example–想實現createorder()最直接的方法就是order service利用REST同步調用其他服務，然而使用REST會降低order service的可用性若是其他服務不能使用了Order Service 將無法再產生任何order
  • https://medium.com/@jinghua.shih/%E7%AD%86%E8%A8%98-rest-%E5%88%B0%E5%BA%95%E6%98%AF%E4%BB%80%E9%BA%BC-170ad2b45836(REST 介紹)

    Maintaining data consistency across services

  • 維持多個服務的資料一致性將會是巨大的考驗
  • 傳統作法 two-phase,commit-based,distributed transaction management mechanism
  • CH4會介紹為什麼上述的方法並不是合適用在現今的應用程式中

    Obtaining a consistent view of the data

  • 如何將多種資料庫的資料同時展示在同一個服務當中，就算解決資料庫一致性的問題但還是會出現這樣的問題。

    God Classes preventing decomposition

  • God Class 當一個物件在系統當中控制太多其他物件，變成了可以做任何事情的class(class that does everything)
  • Example - FTGO 的Order class

    Summary

  • 應用程式結構(建築)決定了maintainability,testability,and deployability
  • 微服務架構是一種達成高度maintainability,testability,and deployability的設計風格
  • Service在微服務中是以business concerns包含business capabilities and subdomain而非technical concerns
  • 兩種decomposition pattern
    • 以business capability作為分解核心
    • 以subdomain利用DDD(domain driven design)的概念進行分解
