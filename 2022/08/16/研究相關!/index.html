研究貢獻

1/22

  • 本研究主要貢獻有五點，首先由於大多數關於ES&CQRS的論文中皆未寫到如何除裡分散式的service若是遇到failure時該如何應對。以Mircoservice架構所設計出來的系統會因為service皆由API以event的形式進行傳遞所以各service之間是無法有效偵測其健康狀態，本研究所設計的是針對以ES&CQRS分散式系統為基礎的錯誤偵測及錯誤回復的pattern，watchdog模組將可以有效的監控所有service的健康狀態甚至能做到自動重啟修復等功能。

  • 第二常見的監控系統多數是中心化的，考慮到中心化的監控系統有可能也會有失效的可能性，在本研究在設計上新增了supervivor watchdog機制有效避免中心化的單一失效問題。

  • 第三使用UDP作為watchdog傳送訊息的媒介，有效防範若MQTT失效時監控系統仍然可以正常運作

    • UDP 訴求的是傳輸輕量化(heatbeats是額外的效能損失)，降低網路頻寬(並不是我發明的不能變成一個貢獻)

  • 第四基於event driven architecture中在面對多數使用者進行交易時將可能造成交易問題寫入漏失的情況發生(3.4會詳細描述)，而本研究設計出來的watchdog機制將會攔截所有因為版本不符而drop的event並自動重新執行。

  • 第五由於本研究主題是系統的穩定性，在面臨系統因為service失效而無法使用的情況下，必須做一些防護措施告知使用者端系統可能出現問題。針對這樣的問題我將會在MQTT傳送command指令前設計Circuit Breaker解決此問題。

    • CB解決了甚麼問題

      2/24

  • Detection

    • Service failure
    • MainDog failure
    • Race condition Data drop

  • Recovery

    • Data drop resend
    • Surpvivior watchdog recovery
    • Service recovery or restart to replace
    • Circuit Breaker recover

  • 本研究的貢獻主要有三點

    • 第一點是關於偵測錯誤
      • 透過WatchDog監控所有Service健康狀態
      • MainDog失效偵測 (使用UDP降低網路頻寬)
      • Race Condition 交易未寫入情況偵測
    • 第二點是關於修復功能
      • 交易未寫入的重新寫入功能
      • MainDog自我修復功能(共識演算法)
      • Service失效的自動修復or取代功能
    • 第三點加入Circuit Breaker 偵測系統的可使用性並且防止client端一直發送無效請求

實驗架構設計

Software Failure Rate VS Recovery time

  • 創建50個service設計類似chaos monkey隨機出錯
  • 是否能成功自動修復
  • 系統的錯誤程度(失效節點數量)VS回復速度

    Node number VS Recovery time

  • 10-1000 節點逐漸增加
  • 觀察watchdog分別透過UDP及MQTT溝通時的效能變化
  • 指定破壞MainDog
  • 針對watchdog 共識演算法測試對節點的修復時間及效能

    Command 數量 VS respond time –UDP/MQTT

  • 10-10000 Command 接近同時傳送，觀察系統處理效能(下的command皆對同一個筆資料改寫)
  • 針對race condition problem的回復機制VS回應時間
  • 測試Circuit breaker的功能(承受多少command指令後斷開後不會出現drop command)
